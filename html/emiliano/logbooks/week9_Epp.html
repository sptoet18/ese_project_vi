Week 9

Overview

This week was focused on the tiered priority queues for the elevator's Finite State Machine on the Supervisory Controller. The goal was to make sure requests from the car controller, the floor controllers, and the website are all serviced in the correct order (Car > Floor > Website), and to capture status videos of the queue working on the real elevator.

Data Structures for the FSM Queue

Each queue just holds floor numbers (1, 2, or 3) waiting to be served, in the order they were requested (FIFO — first in, first out). Three separate queues, one per request source, is what makes this a tiered priority queue: instead of one queue sorted by a priority number, we have three plain queues and always drain the highest-priority one first.


std::queue<int> carQueue;      // priority 1 (highest)
std::queue<int> floorQueue;    // priority 2
std::queue<int> websiteQueue;  // priority 3 (lowest)
Priority Source Queue Entry point
1 (highest) Car controller (0x200) carQueue fsmProcessMessage()
2 Floor controllers (0x201–0x203) floorQueue fsmProcessMessage()
3 (lowest) Website (MySQL) websiteQueue fsmPollWebsite()

How a Request Flows Through the Queues

Step 1 — Getting a request into a queue: fsmProcessMessage()

Every time a CAN message is received (via pcanFsmRxPoll() in the main loop), it's handed to fsmProcessMessage(fsm, id, data), which does a switch(id):


case ID_F1_TO_SC: case ID_F2_TO_SC: case ID_F3_TO_SC:
    floor = ...;
    if (!queueContainsFloor(fsm->floorQueue, floor)) {
        fsm->floorQueue.push(floor);
    }
    break;

case ID_CC_TO_SC:
    floor = FloorFromHex(data);
    if (!queueContainsFloor(fsm->carQueue, floor)) {
        fsm->carQueue.push(floor);
    }
    break;
  • The CAN ID tells us which queue the request belongs in (Car Controller 0x200 → carQueue; floor controllers 0x201/0x202/0x203 → floorQueue).
  • queueContainsFloor() (explained below) is a dedup guard — no point pushing "floor 2" twice if it's already waiting in that queue.
  • The website source doesn't come through CAN at all — it's handled separately by fsmPollWebsite() (step 2), which pushes into websiteQueue the same way.

At this point, a request is sitting in a queue but nothing has happened yet — it's just waiting to be picked up.

Step 2 — Website requests get into their queue differently: fsmPollWebsite()

Called once per fsmStep() tick, throttled to once a second:


int dbFloor = db_getFloorNum();
if (dbFloor == fsm->lastDbFloor) return;   // no change, nothing to do
fsm->lastDbFloor = dbFloor;

if (dbFloor >= 1 && dbFloor <= 3 && !queueContainsFloor(fsm->websiteQueue, dbFloor)) {
    fsm->websiteQueue.push(dbFloor);
}
  • It reads the MySQL currentFloor field (same field the original case 3 menu option read).
  • Compares against lastDbFloor — this is how it tells "the website changed the value" apart from "nothing changed since I last checked."
Important trick: when the elevator itself arrives somewhere, fsmArrive() also writes to that same DB field (see step 5) and immediately updates lastDbFloor to match. That way, the FSM's own position update is never mistaken for a fresh website request — only a change that didn't come from fsmArrive() gets queued.

Step 3 — Deciding where to go next: fsmPeekPriority()

This is the heart of the "priority" behavior. It only runs when the elevator is stopped and the door is closed:


tmp = fsm->carQueue;
while (!tmp.empty()) {
    int f = tmp.front(); tmp.pop();
    if (f != currentFloor) return f;   // found a usable car request — done
}

tmp = fsm->floorQueue;
while (!tmp.empty()) { ... same check ... }   // only reached if carQueue had nothing usable

tmp = fsm->websiteQueue;
while (!tmp.empty()) { ... same check ... }   // only reached if carQueue AND floorQueue had nothing
  • It copies each queue into a temporary (tmp) so peeking doesn't destroy the real queue — we're just looking, not committing yet.
  • It walks carQueue first. If it finds any entry that isn't the current floor, that's the answer — floor requests and website requests are never even looked at.
  • Only if carQueue is completely empty (or only contains the current floor) does it move on to floorQueue, and only if that's empty too does it fall through to websiteQueue.
  • Returns 0 if literally nothing is pending anywhere — meaning the elevator just stays put with the door closed.

This strict "don't even look at the next tier unless this one is empty" ordering is exactly what makes it Car > Floor > Website rather than, say, a weighted scoring system.

Step 4 — Committing to a target and cleaning up: inside fsmStep()

Once fsmPeekPriority() returns a real target floor:


int target = fsmPeekPriority(fsm, currentFloor);
if (target != 0 && target != currentFloor) {
    fsmConsumeRequestsForFloor(fsm, target);   // remove it from ALL THREE queues
    fsm->targetFloor = target;
    fsm->state = STATE_MOVING;
    pcanFsmTx(ID_SC_TO_EC, HexFromFloor(target));
}

fsmConsumeRequestsForFloor(fsm, floor) calls purgeFloorFromQueue() on all three queues for that floor number:


static void purgeFloorFromQueue(std::queue<int> &q, int floor) {
    std::queue<int> kept;
    while (!q.empty()) {
        int f = q.front(); q.pop();
        if (f != floor) kept.push(f);   // keep everything except the floor we're now serving
    }
    q = kept;
}
Why purge from all three, not just the queue it came from? Because if floor 2 was requested from inside the car and someone pressed the floor-2 call button and the website also asked for floor 2 — one trip to floor 2 satisfies all three. Otherwise the elevator would immediately think it still owes a "floor 2" trip to whichever queue(s) weren't the one we peeked from.

Step 5 — Arriving and closing the loop: fsmArrive()


fsmConsumeRequestsForFloor(fsm, floor);   // safety net — purge again in case anything slipped in mid-trip
db_setFloorNum(floor);                    // tell the website where we really are
fsm->lastDbFloor = floor;                 // prevent this write from re-triggering fsmPollWebsite()

This closes the loop back to the database, and re-arms the door timer (handled just above this in the function) so the whole cycle — enqueue → peek → consume → move → arrive → door open/close → peek again — repeats.

The Two Small Helpers Tying It Together

  • queueContainsFloor(queue, floor) — copies the queue, walks it, checks if the floor is already in there. Used purely to avoid double-pushing the same floor into a queue that already has it pending.
  • purgeFloorFromQueue(queue, floor) — rebuilds the queue keeping everything except the given floor. Used both when a target is claimed (step 4) and on arrival (step 5).

Both work by copying/rebuilding rather than mutating in place, because std::queue doesn't support random access or removal from the middle — with only 0–3 tiny integers per queue, this is cheap and keeps the logic simple.

Challenges

It was a difficult task to get the priority behavior working, but it worked. The hardest part was making sure a single trip satisfies every queue that asked for that floor, and that the elevator's own position updates never get mistaken for fresh website requests.
