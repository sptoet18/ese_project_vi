Week 10

Overview and Objectives

The work carried out during Week 10 concentrated on the data-management layer of the supervisory controller, which is responsible for bridging the local-area network (web interface) and the CAN network within the elevator system. The principal objective was to design and implement the relational database that records the operational state of each network node, and to establish the foundation required for the web interface to read from and write to that database.

To meet this objective, the relational schema was defined, the core tables were created within a MySQL instance administered through phpMyAdmin, and the supporting SQL procedures for creating, describing, and modifying tables were formalised. In parallel, the basic functions required to interface the web front end with the database were developed. The sections that follow document the schema design, the SQL methodology adopted, the key constraints applied, the week's day-to-day progress, and the outstanding challenges identified for subsequent weeks.

Database Schema Development

The database was structured around a primary table describing the elevator network as a whole, together with a related sub-network table describing the individual CAN nodes. The design applies a parent–child relationship between these two tables so that the integrity of node records is enforced at the database level.

Primary table — elevatorNetwork

The elevatorNetwork table serves as the master record of the system. Each entry captures a timestamped snapshot of a node together with its status and floor information. The nodeID field is declared as the primary key and is configured to auto-increment, so that every record receives a unique identifier automatically. All fields are declared NOT NULL to guarantee that no incomplete records are admitted.

Sub-network table — CAN_subNetwork

The CAN_subNetwork table records the state reported by each CAN node. Its CAN_nodeID field is defined as a foreign key that references the nodeID of the elevatorNetwork table. As a consequence, a row may only be inserted into this child table when a corresponding parent record already exists, which preserves referential integrity across the two tables.

SQL Methodology

A consistent set of SQL procedures was adopted for creating and maintaining the tables. These procedures are recorded below both as a reference for the remainder of the project and as evidence of the commands exercised during this week's work.

Establishing a session and selecting a database

Database sessions were opened from the command line by authenticating as the project user, after which the working database was selected before any table operations were performed.


# Log in to the MySQL server as the project user
mysql -u Emiliano -p

# Select the database to operate on
USE [database name];

Creating a table

Tables were created by declaring each field with its data type and null constraint, and by specifying the InnoDB storage engine so that foreign-key relationships could be enforced.


CREATE TABLE [table name] (
    date            DATE             NOT NULL,
    nodeID          TINYINT UNSIGNED NOT NULL
    -- ... remaining fields ...
) ENGINE = InnoDB;

Inspecting a table

The structure of any table was verified using the describe command, which lists each field together with its type, null setting, key, and default.


DESC [table name];

Modifying an existing table

Existing tables were altered to add new fields or to redefine the properties of a field. New fields were positioned relative to existing ones, and a field was promoted to an auto-incrementing primary key by modifying its definition.


-- Add a field at a chosen position
ALTER TABLE elevatorNetwork
    ADD [field] [type] AFTER [field];

-- Promote a field to an auto-incrementing primary key
ALTER TABLE [table name]
    MODIFY [field] [type] UNSIGNED NOT NULL
    AUTO_INCREMENT PRIMARY KEY;

Key Constraints

Three categories of key were applied during the schema design. Each enforces a distinct guarantee about the uniqueness or the relational integrity of the data.

  • Primary Key (one per table) — a primary key defines a field for which every row must hold a unique value. It functions much like the index of an array, providing an unambiguous handle to each record. In this schema, nodeID serves as the primary key of the elevatorNetwork table.
  • Foreign Key (parent ↔ child) — a foreign key links a child table to a parent (reference) table. It prevents a row from being inserted into the child table unless a matching record already exists in the parent, thereby preserving referential integrity. The InnoDB engine is required for this behaviour, so tables using foreign keys are declared with ENGINE = InnoDB.
  • Unique Key (no repeats) — a table may hold only one primary key; where additional fields must also be free of duplicate values, they are declared as unique keys. A unique constraint guarantees that no two rows share the same value in that field.

-- Declare a foreign-key relationship
FOREIGN KEY ([field]) REFERENCES [parent table]([field])
ENGINE = InnoDB;

-- Enforce uniqueness on an additional field
ADD UNIQUE KEY ([field]);

Challenges

Two principal challenges were identified during the week. Both concern the coordination between the distributed controllers and the timely propagation of state to the database.

Challenge 01 — Inter-controller coordination. Determining the correct mechanism by which the car controller communicates with, and updates, its door state in accordance with the commands issued by the supervisory controller. Resolving this requires a clear definition of the message exchange between the car controller and the supervisory controller.
Challenge 02 — Dynamic database updates. Enabling the database to be updated dynamically as the system runs. This will require the adoption of socket programming so that live state changes can be pushed to the database without manual intervention.

Next Steps

  • Define the message protocol governing communication between the car controller and the supervisory controller for door-state updates.
  • Study and apply socket programming to allow the database to be updated dynamically during operation.
  • Extend the web interface to perform full read and write operations against the completed schema.
